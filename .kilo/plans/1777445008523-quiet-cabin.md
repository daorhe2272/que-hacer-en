# Plan: Filter Out Events > 60 Days in Future

## Context
The LLM event extraction pipeline already discards past events via `isEventInPast()` in `packages/api/src/utils/event-processor.ts`. We need to add an equivalent filter that discards events scheduled more than **60 days in the future** (counting from today in Colombia timezone).

## Proposed Changes

### 1. `packages/api/src/utils/event-processor.ts`
- **Add** a new helper `isEventTooFarInFuture(eventDate: string): boolean` that mirrors the Colombia-timezone logic in `isEventInPast` but checks if the event date is > 60 days after today.
- **Insert** the new check inside `processExtractedEvents()` right after the existing past-date check, with a descriptive skip message (e.g., `Evento muy lejano (${eventDate})`).
- Keep both checks as separate, clearly named functions for readability.

### 2. `packages/api/tests/event-processor.spec.ts`
- **Add** test cases for the new filter:
  - Event 61 days in the future → skipped.
  - Event exactly 60 days in the future → accepted (if `>` is used).
  - Event 1 day in the future → accepted.
- Update any mock dates that might accidentally trigger the new filter (the current test uses `2027-06-15`, which is fine, but we should be explicit).

## Rationale
- **Consistency**: Reuses the existing Colombia-timezone (`America/Bogota`) date math so server timezone never affects the result.
- **Simplicity**: Two small, pure helper functions are easier to test and reason about than one combined validator.
- **User value**: Prevents the catalog from being cluttered with events that are too far out to be relevant.

## Questions / Open Points
- Should the 60-day threshold be hard-coded or configurable via an environment variable (e.g., `MAX_FUTURE_EVENT_DAYS`)?
- Should we keep existing tests that use `2027-06-15` (currently > 60 days away) or update the test date to something within the window so the "valid event" test stays valid under the new rule?
